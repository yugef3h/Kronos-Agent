# Monorepo 模块边界 & 依赖规范 Checklist
适用：Turbo / Nx / pnpm workspace / yarn workspace monorepo

## 核心原则
1. 分层严格隔离：应用层(apps)、公共包(packages)、内部私有模块(internal) 职责清晰
2. 依赖单向：业务应用依赖公共包，公共包禁止反向依赖业务应用
3. 所有跨模块调用必须走**公开导出入口**，禁止直接引用内部文件路径
4. 公共包只做通用能力：工具、常量、类型、通用逻辑，不承载业务专属逻辑
5. 模块接口稳定：对外导出尽量收敛，不随意暴露内部实现

## 检查项（评审必逐条核对）
### P0 致命违规
- 应用层绕过模块导出，直接引用 packages 内部子文件、私有函数
- 公共包反向依赖 apps 业务层代码
- 跨模块硬编码路径引用，不使用 workspace 声明依赖
- 业务逻辑下沉到通用公共包，污染底层职责
- 删除/修改公共导出接口，未做向后兼容、未兼容老调用方

### P1 高风险
- 一个包职责过载，同时承载通用工具 + 专属业务
- 类型、枚举、常量散落各应用，未收敛到 packages/common
- 模块导出过多，内部实现全部暴露，无封装
- 跨模块状态隐式耦合，无统一抽象层

### P2 优化建议
- 包拆分粒度不合理，可拆为更小单一职责子包
- 缺少模块入口 index.ts 统一收敛导出
- 内部工具未沉淀到公共包，各应用重复拷贝

## 评审要求
1. 跨模块改动必须按「包/模块」分组独立评审
2. 先评估**架构边界、依赖方向、导出合规性**，再评审业务细节
3. 边界违规优先标记 P0/P1，优先整改，再处理代码风格、冗余等低优先级问题
4. 大体积跨模块 diff：先全局结构概览，再按包分批评审，最后汇总整体架构风险
