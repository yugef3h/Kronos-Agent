# 外卖仿真 Agent 可行性计划（不接真实交易能力）

## 1. 范围与目标

### 1.1 范围边界
- 不接入真实淘宝闪购、支付宝、高德等生产系统。
- 不发起真实交易、真实支付、真实履约。
- 保留 JWT 鉴权与统一 API 网关约束。

### 1.2 交付目标
- 将外卖流程从“前端本地 mock”升级为“后端可编排响应 + 前端可视化状态机”。
- 建立可观测事件流：规划、工具、推理、卡片指令、失败降级。
- 提供可回放、可测试、可扩展的仿真链路。

## 2. 可行性结论
- **技术可行**：现有 SSE、timeline、JWT、会话存储已具备基础能力。
- **实现成本可控**：优先复用现有聊天主链路与 tool 事件结构。
- **风险可管理**：通过“规则层兜底 + 模型层增强 + 失败降级”保障稳定性。

## 3. 目标架构

### 3.1 后端响应三层
1. Scenario 引擎（规则/剧本）
- 负责流程状态推进与关键字段真值。
- 输出可重复事件，作为默认兜底层。

2. 数据快照层（可选增强）
- 提供店铺、价格区间、评分、配送时效等静态或半静态数据。
- 不做在线实时抓取，减少波动。

3. 模型表达层（Doubao Stream）
- 负责解释文案、推荐理由、追问语句。
- 通过 JSON Schema 约束输出格式。

### 3.2 编排原则
- 数值字段从规则/快照层来。
- 文本字段由模型层补充。
- 合并后统一校验再推送给前端。
- 模型异常时回退 Scenario 文案。

### 3.3 提示词编排策略（本次调整）
- 不再只做“是否有外卖意图”的二分类。
- 后端先对用户输入做 LLM 提示词加工，注入明确的角色、任务和输出规范。
- 编排器输出三种动作：
	1. `chat`：普通闲聊（自然语言回复）
	2. `ask_slot`：外卖参数不全，反问补全
	3. `tool_call`：参数完整，生成隐藏工具调用参数
- `tool_call` 不在对话框直接显示，前端仅根据结构化字段静默触发外卖模块。

## 4. 分阶段计划

### Phase 1（1-2 天）：协议与最小链路
- 定义外卖 SSE 事件协议：`intent`, `slot`, `tool`, `card`, `complete`, `error`。
- 新增后端仿真入口（可先独立 endpoint，再并轨主链路）。
- 前端新增 adapter，替换现有 `doubaoMockApi.ts` 调用方式。

**验收**
- 从外卖触发到结算卡片可全程由后端响应驱动。
- 失败时可降级回规则文案，前端不白屏。

### Phase 2（2-3 天）：意图与槽位
- 落地三段式识别：规则短路 + 轻量分类 + 槽位抽取。
- 增加 `confidence` 与 `missingSlots` 输出。
- 增加追问策略，确保低置信度不误入下单流程。
- 接入 LLM 编排器：按“角色 + 任务 + 格式要求”输出 `chat/ask_slot/tool_call` 动作。

**验收**
- 命中率、误判率达到可演示标准。
- 至少覆盖 30 条正负样例测试。

### Phase 3（2-3 天）：工具化与可观测性
- 将地址确认、推荐、优惠、结算拆成可观测工具步骤。
- 与 timeline 面板对齐展示 tool input/output/error。
- 增加 traceId，串联一次完整外卖会话。

**验收**
- 每次流程可回放关键步骤。
- 工具失败可追踪、可重试、可降级。

### Phase 4（1-2 天）：数据快照接入与清理
- 把前端固定商品数据迁移到后端快照源。
- 前端仅保留展示 DTO，不存业务真值。
- 清理冗余 mock 常量和散落文案。

**验收**
- 前端不再依赖核心 `MOCK_FOODS/ADDRESS/DISCOUNT` 作为主数据源。
- 文档与测试同步更新。

## 5. 风险与应对
1. 模型返回不稳定
- 应对：Schema 校验 + 规则兜底 + 重试上限。

2. 爬虫/快照合规风险
- 应对：仅采集公开、非敏感、允许范围数据；必要时使用自建样例库替代。

3. 前后端协议漂移
- 应对：增加契约测试与版本字段 `schemaVersion`。

4. 演示体验抖动
- 应对：关键节点先返回可渲染骨架，再补全流式文案。

## 6. 质量门禁
- 单元测试：意图识别、槽位抽取、状态推进、金额计算。
- 契约测试：SSE 事件结构与字段必填校验。
- 集成测试：外卖触发到支付完成的 Happy Path 与降级 Path。
- 文档门禁：接口协议、状态机图、异常码说明。
- 新增：提示词编排测试（隐藏工具标签解析、可见回复剥离、异常降级）。

## 7. 团队协作建议
- 后端先交付协议 mock server，前端先改 adapter。
- 每个阶段结束固定回归样例集，防止交叉回归。
- 在 PR 模板中强制填写：新增事件、回退策略、测试覆盖情况。

## 8. 意图解析建议（实现优先级）
1. 先规则：高精度关键词与短语模板。
2. 后分类：轻量模型打分 `takeout_order/takeout_recommend/non_takeout`。
3. 再槽位：地址、预算、偏好、时间。
4. 最后追问：只追问当前最影响决策的一个槽位，减少交互轮次。

建议指标：
- 意图识别准确率 >= 92%
- 误触发率 <= 3%
- 关键槽位完整率 >= 85%
- 首次可推荐率 >= 80%
