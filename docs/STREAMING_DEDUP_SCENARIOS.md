# 流式输出重复问题说明（两种场景对比）

## 背景

在 SSE 流式输出里，前端通常会把每个 `content` 片段做追加渲染：

- 新片段 1：`中华`
- 新片段 2：`人民`
- 新片段 3：`共和国`

最终显示：`中华人民共和国`

如果后端和前端对“片段语义”理解一致（都按增量），不会重复。
如果语义不一致，或者事件被重复消费，就会出现：
`中华中华人民共和国的人民共和国...`

---

## 场景 A：旧 Demo（不会重复）

对应实现：

- `node/server.js`
- `src/app.js`

### 为什么不会重复

1. 后端只发送增量字符
- `node/server.js` 的 `generateStream` 中，回复先 `split('')`。
- 每次 `yield` 都是一个新字符（天然 delta）。

2. 断点续传是“从断点继续”，不是“重发累计文本”
- 通过 `lastEventId` 控制循环起点：`for (let i = lastEventId; i < tokens.length; i++)`。
- 不会把已经发过的片段重新发给同一轮渲染。

3. 前端只做简单追加就够了
- 因为每个片段天然是增量，`last += chunk` 不会产生重叠。

### 场景结论

旧 Demo 的协议语义非常简单：
**一条 SSE content 事件 = 一段新增内容（delta）**，因此几乎不可能出现重复拼接。

---

## 场景 B：当前工程（曾出现重复）

对应实现：

- 后端：`apps/server/src/services/langchainChatService.ts` + `apps/server/src/services/streamService.ts`
- 前端：`apps/web/src/components/ChatStreamPanel.tsx`

### 为什么会出现重复（历史问题）

主要有两类风险：

1. 模型 chunk 语义不稳定
- 某些上游会返回“累计文本”，例如：
  - `中华`
  - `中华人民共和国`
  - `中华人民共和国的首都是北京。`
- 如果前端仍按增量追加，就会把重叠部分重复拼接。

2. 前端重复消费事件
- 网络抖动/重连、并发请求未隔离、旧流未终止，都可能让同一轮消息被写入两次。
- 即便后端是增量，重复消费同一事件也会重复显示。

### 你当前抓包说明了什么

你给到的接口片段是：

- `中华`
- `人民`
- `共和国`
- `的`

这组数据本身是增量（不是累计）。
因此当时重复更可能是前端消费层问题，而非该次后端分片内容本身。

---

## 修复策略（当前采用）

### 1. 后端层：统一增量语义

新增：`apps/server/src/services/streamDelta.ts`

- `normalizeStreamDelta(previous, current)`
- 当 `current` 以前缀包含 `previous` 时，只取新增后缀。

作用：
- 上游若返回累计片段，也会在服务端被转换成稳定 delta 再发给前端。

### 2. 前端层：防重复消费

在 `apps/web/src/components/ChatStreamPanel.tsx` 增加两层保护：

1. 单活动流控制
- 新请求发起前，先 `abort` 旧流。
- 使用 `activeRequestIdRef` 忽略过期请求回调。

2. 同轮事件去重
- 用 `lastSeenEventId` 去重：
- 如果 `payload.eventId <= lastSeenEventId`，直接忽略。

作用：
- 防止重连重放、并发回调、旧请求残留写入导致的重复显示。

---

## 两种场景的本质差异

### 旧 Demo

- 协议简单：固定增量
- 运行路径简单：单路流
- 客户端逻辑简单：直接追加

### 当前工程

- 协议更复杂：上游 chunk 语义可能变化（增量/累计）
- 链路更复杂：模型流 + fallback + 前端重连机制
- 需要“双保险”：后端归一化 + 前端去重隔离

---

## 建议的长期规范

1. 协议规范化
- 明确约定 `content` 事件只承载 delta。
- 若需要累计文本，使用独立字段（例如 `fullText`），避免语义混用。

2. 事件幂等
- 每个事件带严格递增的 `eventId`。
- 前端默认按 `eventId` 去重，作为基础防线。

3. 请求隔离
- 任意时刻只保留一个活跃流。
- 新请求必须显式终止旧请求。

4. 可观测性
- 保留 timeline 日志和 fallback reason。
- 问题复现时先看：
  - 后端输出是否是 delta；
  - 前端是否重复消费同一 `eventId`。

---

## 复盘一句话总结

旧 Demo 不重复，是因为它天然只发增量；
当前工程会重复，是因为真实链路更复杂，需要同时处理“分片语义归一化”和“事件消费去重”。

## 市场成熟方案与 npm 选型建议

## 结论

目前行业里“优雅且稳定”的做法，不是依赖单一去重包，而是采用：

- 协议层约束（明确 delta 语义）
- 客户端幂等处理（eventId 去重 + 单活动流）
- 可观测性（timeline + fallback reason）

在此基础上，优先选成熟 SDK/库来减少自研状态管理成本。

## 常见方案对比

1. `ai`（Vercel AI SDK）
- 特点：流式消息状态管理、UI 集成、协议处理较完整。
- 适合：希望快速得到稳定流式体验，减少前端样板代码。
- 注意：后端数据协议最好与其推荐格式对齐，迁移成本更低。

2. `@microsoft/fetch-event-source`
- 特点：支持 POST、Header、Abort、重连控制，比原生 EventSource 更适合业务链路。
- 适合：像当前项目这样有 JWT、自定义协议、SSE 细节控制需求的场景。
- 现状：本项目已采用，继续沿用是合理路线。

3. `eventsource-parser`
- 特点：专注 SSE 解析，处理粘包/半包等底层细节。
- 适合：服务端或中间层需要自己处理原始流时，提升解析可靠性。

## 对当前项目的建议路线

### 路线 A：保持现有架构（推荐）

- 保留 `fetch-event-source`。
- 继续坚持：后端 delta 归一化 + 前端 eventId 去重 + 单活动流。
- 成本低、可控性强、和当前代码最兼容。

### 路线 B：逐步迁移到 `ai`

- 用 `ai` 托管前端流式消息状态。
- 后端保留现有鉴权与路由，逐步对齐数据协议。
- 适合后续要扩展工具调用轨迹、多模型对话 UI 的阶段。

## 实践建议（可直接执行）

1. 在接口契约文档中明确：`content` 字段默认只传 delta。
2. 前端保留 eventId 去重，不依赖“上游一定正确”。
3. 每次回归测试加入“累计片段输入”与“重复 eventId 重放”用例。
4. 问题排查优先级：先看服务端输出语义，再看前端是否重复消费。
